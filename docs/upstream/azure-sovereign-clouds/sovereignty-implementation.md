# Sovereign design and implementation considerations

Building sovereign and resilient cloud environments requires deliberate choices across platform architecture, operations, data handling, and governance. This article provides a structured set of design and implementation considerations, organized by domain and tiered by control level (Baseline, Elevated, and Advanced), to help organizations apply the right sovereignty and resilience controls for each workload by using the Microsoft Sovereign Cloud portfolio.

## Foundational considerations for sovereign and resilient solutions

Sovereignty and resiliency requirements depend on the workload and are based on risk. Not all workloads require the same level of sovereignty or resilience. Consider the following principles:

- Assess workloads individually based on data sensitivity, regulatory exposure, and business criticality. Apply controls that address risk management, legal compliance, costs, innovation, scale, and control accordingly.
- Treat sovereign design requirements as complementary, not exclusive.
- Combine technical, operational, and contractual controls into a comprehensive approach that strengthens sovereignty and resilience across cloud environments.

The Microsoft Sovereign Cloud integrates sovereignty controls directly into public, hybrid, and private cloud models. It gives organizations flexibility instead of forcing a choice between isolation and hyperscaling cloud benefits. This integration allows organizations to maintain access to innovation while meeting regulatory and geopolitical requirements. 

This risk-based, layered approach enables organizations to apply stronger controls where required, while allowing lower-risk workloads to benefit from broader public cloud capabilities and innovations.

For more information, see [Microsoft Sovereign Cloud overview](microsoft-sovereign-cloud.md) and [Implementing sovereign workloads](public/overview-implement-workloads.md).


## Sovereignty and resilience areas and considerations

This section provides design and implementation considerations to enhance sovereignty and resilience for cloud and hybrid environments. The considerations apply to [Azure](public/overview-implement-workloads.md), [Microsoft 365](public/microsoft-365-sovereign-controls.md), [Dynamics 365 Business Central](public/access-controls-d365-business-central.md), and [Dataverse and Power Platform](public/sovereign-controls-dataverse-power-platform.md).

The considerations aren't prescriptive or mutually exclusive. Adapt them to your organizational context and combine them intentionally to form a coherent, end-to-end sovereignty and resilience posture for each workload. Each consideration is organized along two dimensions.

The first dimension is the domain it belongs to:

- [Platform and deployment](#platform-and-deployment)
- [Operational controls](#operational-controls)
- [Data controls](#data-controls)
- [Governance, risk, and compliance](#governance-risk-and-compliance) 

The second dimension is the level of control it represents:

- **Baseline**: fundamental controls to implement for all environments.
- **Elevated**: controls for more stringent requirements.
- **Advanced**: controls for the most sensitive workloads or those subject to the strictest regulations.

Start with the baseline considerations and move into higher tiers where needed. A practical starting point for implementation is the [Sovereign landing zone (SLZ)](public/overview-sovereign-landing-zone.md), which provides reference architectures and guardrails aligned to these levels.

For more information, see [Controls and principles in Sovereign Public Cloud](public/overview-controls-principles.md) and [Implement controls and principles in SLZ](public/implement-controls-principles.md).

## Platform and deployment 

Sovereign solutions must provide [technological independence](technological-independence.md), strong security controls, resilience, and alignment with recognized standards. These solutions require clear control over data location, encryption, and processing, backed by oversight and transparency.

In practice, these requirements mean designing for portability, interoperability, and verifiable control by using open standards, minimizing provider-specific lock-in, enforcing policy as code guardrails, and maintaining clear exit and recovery paths. Together, these capabilities safeguard data, preserve operational and technological independence, and support evolving regulatory requirements.

| Consideration | Description |
|----|-------------|
| **Baseline** ||
|Open Source and Industry Standards| • Adopt open source technologies and open interfaces where feasible to improve portability, interoperability, and long-term technological independence.<br><br>• Prefer architectures based on broadly adopted standards (for example, containers, Kubernetes, OpenAPI, OAuth 2.0/OIDC, and standard data formats) so workloads and integrations can be moved or replatformed with limited refactoring.<br><br>• Avoid hard dependencies on proprietary runtime features unless there's a clear business justification. When such dependencies are required, document them explicitly and define mitigation plans (for example, abstraction layers, data export paths, and alternative deployment targets).<br><br>• Maintain reusable IaC templates and CI/CD pipelines that you can adapt across environments, and periodically validate portability through migration or exit rehearsals.<br><br>For more information, see [Technological independence](technological-independence.md), [Open source on Azure](https://azure.microsoft.com/solutions/open-source/). |
|Network Architecture and Controls | • Implement strict network isolation and hardened network configurations.<br><br>• Deploy workloads into Azure Virtual Networks (VNet) and use Private Endpoints or Service Endpoints for Azure PaaS services to prevent exposure to the public internet.<br><br>• Apply network security groups (NSGs) and Azure Firewall to restrict traffic flows. Enable DDoS protection if needed. <br><br> • Use Azure API Management to centrally secure and govern APIs by enforcing authentication and policy controls. <br><br>• Follow Azure’s baseline network security configurations in the Sovereign Landing Zone reference to ensure only approved network paths are available.<br><br>For more information, see [Virtual Network](/azure/virtual-network/virtual-networks-overview), [Private Endpoint](/azure/private-link/private-endpoint-overview), [Network Security Groups](/azure/virtual-network/network-security-groups-overview), [Azure Firewall](/azure/firewall/overview), [DDoS Protection](/azure/ddos-protection/ddos-protection-overview), [Azure API Management](/azure/api-management/api-management-key-concepts), [Sovereign Landing Zone](public/overview-sovereign-landing-zone.md). |
|Zero Trust and SFI Enforcement | • Implement Zero Trust Principles for every workload and environment. <br><br>• Include options such as phishing‑resistant MFA using FIDO2 or certificate‑based authentication enforced via Conditional Access, Privileged Identity Management with Just‑in‑Time activation for admin roles, RBAC instead of standing privileges, Managed Identities for workloads, secrets stored and rotated in Azure Key Vault, governance enforced with Azure Policy and management groups.<br><br>For more information, see [Zero Trust overview](/security/zero-trust/zero-trust-overview), [Entra ID Multifactor Authentication](/entra/identity/authentication/concept-mfa-howitworks), [Conditional Access](/entra/identity/conditional-access/overview), [Privileged Identity Management](/entra/id-governance/privileged-identity-management/pim-configure), [Azure Key Vault](/azure/key-vault/general/overview), [Azure Policy](/azure/governance/policy/overview). |

|Policy-as-Code and Compliance Automation | • Integrate policy enforcement and compliance scanning into the Infrastructure-as-Code CI/CD process.<br><br>• Use Azure Policy and initiatives to define sovereign compliance rules (for example, resources must be in approved regions, certain SKU, or feature usage is disallowed). Embed these checks in the deployment pipeline so that any noncompliant infrastructure or configuration is flagged or blocked before provisioning. <br><br>• Continuously scan existing resources for compliance drift using the Azure Policy compliance dashboard. Use Microsoft Purview for data classification, lineage, and data governance controls.<br><br>For more information, see [Azure Policy](/azure/governance/policy/overview), [Policy as Code workflows](/azure/governance/policy/concepts/policy-as-code), [Policy initiatives](/azure/governance/policy/concepts/initiative-definition-structure), [Implement controls and principles in SLZ](public/implement-controls-principles.md). |
|Logging and Monitoring |• Enable thorough logging, monitoring, and threat detection across the enterprise-wide solution landscape.<br><br>• Use Azure-native observability services by function: Azure Monitor (including Log Analytics) for metrics and logs, Azure Activity Log for control-plane resource changes, and Microsoft Defender for Cloud for security posture and threat alerts. <br><br>• Ensure that all security-relevant events (for example, sign-in attempts, configuration changes, data access) are captured and retained in a secure log store. <br><br>• Establish dashboards and alerts for anomalous activities to facilitate prompt incident response. <br><br>For more information, see [Azure Monitor](/azure/azure-monitor/overview), [Log Analytics](/azure/azure-monitor/logs/log-analytics-overview), [Microsoft Defender for Cloud](/azure/defender-for-cloud/defender-for-cloud-introduction), [Azure Activity Log](/azure/azure-monitor/essentials/activity-log). |
|Backup and Disaster Recovery | • Implement robust backup and disaster recovery with tested restore procedures in alignment with hold and retention requirements.<br><br> • Use Azure Backup and Azure Site Recovery where appropriate to regularly back up data and VM images. <br><br>• Define Recovery Time Objectives (RTO) and Recovery Point Objectives (RPO) that meet continuity requirements and test them. <br><br>• Conduct periodic disaster recovery drills to ensure backups can be restored and systems fail over to secondary regions successfully. This approach guarantees continuity of service in line with sovereignty expectations for resilience. <br><br>For more information, see [Business Continuity and Disaster Recovery](public/bcdr-impacts-sovereign-controls.md), [Azure Backup](/azure/backup/backup-overview), [Azure Site Recovery](/azure/site-recovery/site-recovery-overview), [Reliability Checklist](/azure/well-architected/reliability/checklist). |
| **Elevated** ||
|High Availability / Mission-Critical Deployment| • Design the solution for high availability, including multizone or multiregion deployment for critical workloads.<br><br>• If the deployment location offers multiple availability zones (or regions), deploy resources redundantly across zones or regions. For example, use an Active/Passive or Active/Active multiregion architecture with Azure Traffic Manager or Front Door for failover. <br><br>• Select appropriate service tiers and SKUs that support availability zones. Many Azure services require Premium or Standard tiers to enable zone redundancy. Consult service-specific reliability guides to identify SKU requirements for zone support.<br><br>• Regularly test failover processes and ensure the solution tolerates zone failures or even regional outages without violating data residency (for example, failover stays within the EU).<br><br>For more information, see [Regions and Availability Zones Architecture](/azure/well-architected/reliability/regions-availability-zones), [Traffic Manager](/azure/traffic-manager/traffic-manager-overview), [Azure Front Door](/azure/frontdoor/front-door-overview), [Service Reliability Guides](/azure/reliability/overview-reliability-guidance). |
| **Advanced** ||
|<p id="hybrid-local"> Hybrid/Local Deployment Support for Sovereign Private Cloud </p>|• Design the solution's architecture to be deployed on sovereign or disconnected environments (Azure Local and Azure Arc) if needed.<br><br>• Package the application in a way that it can run outside the public cloud. For instance, validate that it works on Azure Arc-enabled infrastructure, to accommodate workloads that require local operation in their own local environment (for example, for extreme sovereignty needs). Offer a virtual machine image or container that a customer can run in an isolated network without external dependencies.<br><br>• For local workloads, decide whether to run them hybrid or mostly air-gapped with Azure Local (that is, with partial internet connectivity), or permanently disconnected with Azure Local Disconnected Operations.<br><br>• Design identity and access so authentication and authorization operate across environments. Consider integration with cloud identity when connected, while providing a local/private IdP and IAM path to operate independently when disconnected. <br><br>For more information, see [Sovereign Private Cloud](private/overview/sovereign-private-cloud.md), [Azure Arc](/azure/azure-arc/overview), [Azure Arc-enabled servers](/azure/azure-arc/servers/overview), [Azure Arc-enabled Kubernetes](/azure/azure-arc/kubernetes/overview), [Azure Local](/azure/azure-local/overview), [Azure Local Disconnected Operations](/azure/azure-local/manage/disconnected-operations-overview), [Identity and access management for Arc-enabled servers](/azure/cloud-adoption-framework/scenarios/hybrid/arc-enabled-servers/eslz-identity-and-access-management). |



## Operational controls
 
[Operational controls](operational-controls.md) and processes ensure that operations are secure, transparent, and accountable. This dimension covers how you detect and manage incidents, handle administrative access, and control operational data (like telemetry and support logs) by using enterprise-wide [control principles](public/implement-controls-principles.md). 

> [!NOTE]
> Treat operational controls from day one as minimum operating hygiene, so every sovereign environment is expected to have access reviews, controlled privileged support access, and active security monitoring from the start.

| Consideration | Description |
|----|-------------|
| **Baseline** ||
| Regular Access Reviews |• Conduct regular reviews of all identities and credentials (users, managed identities, service principals, agents) with access to sensitive data.<br><br>• Use Entra ID Access Reviews to periodically audit user access and privileged role assignments. <br><br>• For each application component, review which service principals and managed identities have data access and remove any that aren't necessary. <br><br>• Implement quarterly (or more frequent) recertification cycles where data owners must confirm who should retain access. This approach ensures the principle of least privilege is maintained over time, closing any gaps where dormant accounts could be exploited to access data.<br><br>For more information, see [Access reviews](/entra/id-governance/access-reviews-overview), [Managed identities](/entra/identity/managed-identities-azure-resources/overview), [Service principals](/entra/identity-platform/app-objects-and-service-principals). |
| Local Approval for Privileged Support Access | • For any required support or maintenance access, enforce privileged access by using just-in-time elevation, local approvals, and session recording.<br><br>• Implement an operational model where if an engineer needs to access a production system or data for support, a localized approval workflow is required. For example, use Azure Privileged Access Management or custom tooling to require a two-person rule - a designated security or customer representative must approve the access request. <br><br>• Record all sessions (by using tools like Azure Bastion session recording or audited PowerShell sessions) and store logs for later review by the customer.<br><br>• For EU/EFTA region deployments, this process is accompanied by the Data Guardian feature and stored in an immutable confidential ledger.<br><br> For more information, see [Privileged Identity Management](/entra/id-governance/privileged-identity-management/pim-configure), [Customer Lockbox](/azure/security/fundamentals/customer-lockbox-overview), [Azure Bastion session recording](/azure/bastion/session-recording), [Data Guardian](public/data-guardian.md), [Confidential Ledger](/azure/confidential-ledger/overview). |
| Continuous Monitoring and Threat Detection | • Use Microsoft Sentinel to aggregate and analyze security logs for threats. <br><br>• Use Microsoft Defender for Cloud and Defender for Endpoint to get alerts on malicious activities or vulnerabilities in the environment. <br><br>• Set up 24/7 monitoring either in-house or with an external Security Operations Center (SOC) service, ensuring that you promptly detect and address any incident. <br><br>For more information, see [Microsoft Sentinel](/azure/sentinel/overview), [Microsoft Defender](/defender-xdr/microsoft-365-defender), [Microsoft Defender for Cloud](/azure/defender-for-cloud/defender-for-cloud-introduction), [Microsoft Defender for Endpoint](/defender-endpoint/microsoft-defender-endpoint). |

## Data controls 

[Data controls](data-controls.md) determine how and where data is stored, processed, and protected. Key aspects include implementing strict data residency, restricting data movement across borders, classifying and tagging data to apply appropriate safeguards, and using strong encryption and key management for data at rest, in transit, and in use.

Design solutions that give customers control over their encryption keys and that use technologies like confidential computing to protect data in use through Trusted Execution Environments (TEE). By applying these controls, organizations can dictate the terms of data handling and provide assurance that data sovereignty and privacy requirements are continuously met.

| Consideration | Description |
|----|-------------|
| **Baseline** ||
| Data Classification and Sovereign Tagging |• Establish clear data classification and tag data to appropriate sovereign control policies, following Sovereign Landing Zone concepts.<br><br>• Define data classification categories, such as Public, Confidential, and Secret, and map them to required controls.<br><br>• Use Microsoft Purview or Azure Information Protection to tag data assets, including files and databases, with these classifications. Then apply automated policies. For example, if data is tagged Confidential, enforce encryption at rest and strict access controls via Azure RBAC.<br><br>• The Sovereign Landing Zone guidance provides the foundation to implement company-specific tagging and policy enforcement for workloads and data in a dedicated management groups hierarchy.<br><br>For more information, see [Microsoft Purview data classification](/purview/ai-microsoft-purview#data-classification-and-ai-interactions), [Data loss prevention (DLP)](/purview/dlp-learn-about-dlp), [Microsoft 365 Rights Management](public/microsoft-365-rights-management.md).|
| Data Residency Enforcement | • Enforce data residency requirements (including telemetry and backups) to remain within the required area, such as national or regional boundaries.<br><br>• Configure all Azure services, including databases, storage, and backup vaults, through infrastructure as code (IaC) and tenant-wide policies to use data centers specific to approved countries/regions. Deny deployments outside those boundaries. Disable geo-redundant replication unless explicitly permitted. Ensure backup vaults and log/monitoring workspaces are pinned to the required region.<br><br>• If applicable, use Azure’s EU Data Boundary to keep all data processing within Europe. Use an EU-based Azure Resource Manager and disable or redirect telemetry to EU endpoints as available, ensuring no data is sent to non-EU locations.<br><br>• For Microsoft 365 workloads, consider Advanced Data Residency (ADR) to commit in-country/region data residency for supported services.<br><br>• Maintain auditable evidence of residency by documenting data flows and periodically reviewing policy compliance and service configurations for drift.<br><br>For more information, see [Azure regions and geographies](/azure/reliability/regions-overview), [EU Data Boundary](/privacy/eudb/eu-data-boundary-learn), [Microsoft 365 data locations](/microsoft-365/enterprise/o365-data-locations?view=o365-worldwide&preserve-view=true), [Azure Policy](/azure/governance/policy/overview), [Azure Storage redundancy](/azure/storage/common/storage-redundancy), [Advanced Data Residency in Microsoft 365](/microsoft-365/enterprise/advanced-data-residency).| 
| AI Data Boundaries | • Enforce restrictions on AI training and inference data so it stays within the required jurisdiction, with logging and evidence of compliance, across all AI surfaces. Examples include Azure AI services and Azure OpenAI/Foundry, Microsoft 365 Copilot/Copilot Chat, and any custom copilots or agents.<br><br>• Use regional or DataZone deployment types to keep processing within the required geography and avoid global deployments when in-country/region processing is required. Pin supporting services such as storage, vector stores, logs, and prompt history stores to approved regions.<br><br>• For Microsoft 365 Copilot, use Advanced Data Residency (ADR), if available. EU users are covered by the EU Data Boundary, which ensures that processing stays within the EU boundary. Validate in-country/region expectations based on the Copilot residency guidance.<br><br>• For fine-tuning or RAG, keep training data and data sources in-region and document the data flow, deployment type, and region settings as audit evidence.<br><br>• Consider a local deployment using Foundry Local (preview) with open source models for the strictest requirements (See the consideration: <a href="#hybrid-local"> Hybrid/Local Deployment Support for Sovereign Private Cloud</a>).<br><br>For more information, see [AI workloads and sovereignty](public/ai-workloads-sovereignty.md), [Data, privacy, and security for Azure Direct Models in Microsoft Foundry](/azure/ai-foundry/responsible-ai/openai/data-privacy), [Deployment types](/azure/foundry/foundry-models/concepts/deployment-types), [Foundry Local](/azure/foundry-local/get-started), [Data, Privacy, and Security for Microsoft 365 Copilot](/copilot/microsoft-365/microsoft-365-copilot-privacy), [Advanced Data Residency in Microsoft 365](/microsoft-365/enterprise/advanced-data-residency).|
| **Elevated** ||
| Bring Your Own Key (BYOK) | • Use customer-managed encryption keys for data at rest. For example, use Azure Key Vault with BYOK, Key Vault Premium, or Managed HSM for critical workloads.<br><br>• Use keys that you control and configure services such as Storage Accounts and SQL Databases to use those keys for encryption at rest. You can revoke access to data or rotate keys, effectively controlling access to your company data. By using this approach, you retain control over the primary encryption keys used to protect data, ensuring that Microsoft services can't routinely decrypt data without customer authorization, while still preserving service availability through tightly controlled break-glass mechanisms.<br><br>For more information, see [Key Management Controls](key-controls.md), [CMK support by Service](/azure/security/fundamentals/encryption-customer-managed-keys-support), [BYOK in Azure Key Vault](/azure/key-vault/keys/byok-specification), [Managed HSM](/azure/key-vault/managed-hsm/overview), [Key recovery management](public/key-recovery-management.md), [Encryption at rest](/azure/security/fundamentals/encryption-atrest).|
| Customer Lockbox for Support | • If you use Microsoft support channels, enable Azure Customer Lockbox so that you must approve any Microsoft engineer’s request to access your data for support.<br><br>• For your own operations team, establish a "customer hold" process where emergency access to production data for troubleshooting requires approval from a designated customer representative or an internal privacy officer and is fully documented.<br><br>For more information, see [Customer Lockbox](/azure/security/fundamentals/customer-lockbox-overview).|
| **Advanced** ||
| Confidential Computing | • Utilize confidential computing for sensitive data and workloads to protect data in use.<br><br>• Deploy sensitive services on Azure Confidential Computing platforms, which use hardware-based Trusted Execution Environments (TEE) to protect data in use.<br><br>For more information, see [Azure Confidential Compute](public/confidential-computing.md).|
| Split-Key or Double-Key Encryption | • Implement cryptographic solutions that use split-key or double-key encryption for the most sensitive data.<br><br>• In a double-key encryption model, data is encrypted with two keys - one managed by the customer outside the cloud and one by the service. Both keys are required to decrypt the data. This approach ensures that data remains inaccessible unless the customer actively participates.<br><br>• For your solution, identify any data that warrants this level of control, such as regulated secrets, and apply a library or service that enforces dual-key access.<br><br>For more information, see [Double Key Encryption](/microsoft-365/compliance/double-key-encryption), [Encryption overview](/azure/security/fundamentals/encryption-overview).|
| External Key Management (preview) |• Use external key management features when available in Azure to store encryption keys entirely outside of the cloud provider's control.<br><br>• Azure is introducing External Key Management (EKM) capabilities where keys can reside in an on-premises HSM or a third-party cloud but still be used by Azure services. This approach provides heightened sovereignty and allows organizations to comply with certain regulations, as keys remain under national or customer control exclusively.<br><br>For more information, see [Key Management Controls](key-controls.md), [External Key Management](public/external-key-management.md).|

## Governance, risk, and compliance

Governance, risk, and compliance (GRC) controls define how sovereign requirements translate into accountable decision-making, enforceable policies, and verifiable compliance. This dimension covers adherence to applicable laws and regulations, alignment with national and sectoral frameworks, governance structures, roles, and oversight mechanisms. Key aspects include lawful handling of data and government access requests, transparency and auditability, executive accountability for sovereignty outcomes, and the integration of sovereignty into risk management, operational processes, and assurance activities.

By embedding sovereignty requirements into governance models and compliance processes, organizations ensure that legal obligations are met by design and continuously enforced, monitored, and independently verifiable. 

| Consideration | Description |
|----|-------------|
| **Baseline** ||
| Sovereignty Accountability Model | • Establish a formal accountability framework for digital sovereignty within the organization. <br><br>• Assign clear executive ownership for the sovereignty program (for example, a senior executive responsible for overseeing compliance with sovereign requirements). <br><br>• Define objectives and key results (OKRs) or similar metrics for sovereignty. Require regular reporting on these metrics to leadership. The accountability model should also outline roles and include sovereign compliance in corporate governance.<br><br>For more information, see [Cloud governance](/azure/cloud-adoption-framework/govern/). |
| Regulatory and Legal Obligations | • Translate applicable laws into clear, auditable controls. This translation process includes identifying the relevant jurisdictions and sector rules, embedding obligations into contracts and operating procedures, and maintaining evidence that requirements are met in practice - not only documented. <br><br>• Key focus areas typically include national data protection, ensuring lawful processing, privacy-by-design, DPAs/sub-processor governance, support for data subject rights, and documented DPIAs where required. It also includes security expectations and industry-specific regulations.<br><br>For more information, see [Microsoft compliance documentation](/compliance/), [Azure compliance offerings](/azure/compliance/offerings/), [Microsoft Products and Services Data Protection Addendum (DPA)](https://www.microsoft.com/licensing/docs/view/Microsoft-Products-and-Services-Data-Protection-Addendum-DPA), [Microsoft Service Agreements](https://www.microsoft.com/en/servicesagreement). |
| Compliance Management and Monitoring | • Implement a compliance management program that continuously monitors adherence to sovereign requirements and provides evidence for audits. <br><br>• Use Azure Policy to enforce compliance with sovereignty controls (for example, data residency, allowed services). <br><br>• Regularly review the compliance dashboard and address any noncompliant resources and compliance drift. <br><br>• For data governance, use Microsoft Purview to classify data and ensure it's handled according to policies. Maintain an internal audit function that periodically reviews the solution’s compliance posture, including configuration, access controls, and operational practices, and produces reports for leadership and regulators.<br><br>For more information, see [Sovereign Policies](public/design-sovereign-policies.md), [Microsoft Purview](/purview/purview), [Compliance Manager](/microsoft-365/compliance/compliance-manager), [Regulated Environment Management Portal](public/regulated-environment-management.md). |
| Third Party Oversight and Transfer Impact Assessments | • Complete and document Transfer Impact Assessments (TIAs) for subcontractors.<br><br>• For any data handled by third parties or moved outside the EU, perform a TIA to evaluate legal risks, and implement supplementary actions as needed. Maintain records of TIAs and require subcontractors to adopt EU Standard Contractual Clauses or equivalent safeguards.<br><br>• Reassess TIAs periodically and on material change events (for example, subprocessor changes, transfer destination changes, or relevant legal/regulatory changes), and retain evidence of review decisions and compensating controls.<br><br>For more information, see [Compliance Manager](/microsoft-365/compliance/compliance-manager), [European Union Model Clauses (Microsoft)](/compliance/regulatory/offering-eu-model-clauses). |
| Exportable Audit Trails and Evidence | • Ensure audit trails and evidence packs are readily exportable for regulators or customers. Set up systems to collect and organize all compliance evidence related to sovereignty. This evidence includes logs of data location (to prove residency), records of access approvals (from Customer Lockbox or privileged access systems), results of drills and audits, and configurations of key controls (like encryption settings). <br><br>• Use tooling to package this evidence when needed – for example, Azure provides compliance reports and you might maintain a repository of configurations (Infrastructure-as-Code templates) and their last audit status. <br><br>• If a regulator requests evidence of sovereignty controls, you should be able to quickly generate an evidence pack demonstrating compliance (for example, an export of Azure Policy compliance results, audit logs, or penetration test results). <br><br>For more information, see [Azure Monitor](/azure/azure-monitor/overview), [Audit Solutions](/purview/audit-solutions-overview). |
| Support Resources and IT Service Management |• Establish a sufficient support model with Microsoft (or equivalent) to allow for rapid response times and high SLA support. <br><br>• Internally, also ensure an on-call process and escalation matrix are established so that any issue gets prompt attention.<br><br>For more information, see [Azure support plans](https://azure.microsoft.com/support/plans/), [IT service management](/azure/cloud-adoption-framework/manage/), [CSP customer support model](/partner-center/customers/customer-support). |
| **Elevated** ||
| Incident Response Playbook | • A documented incident response and security operations playbook are in place, aligned with Azure’s best practices (such as the SFI guidelines).<br><br>• Develop a formal Incident Response Plan detailing procedures for different incident types (security breach, service outage, or data leakage). Align roles and steps with Azure’s Cloud Adoption Framework guidance for incident management. <br><br>• Regularly train the operations team on this playbook and keep it updated.<br><br>For more information, see [Incident management](/azure/cloud-adoption-framework/manage/protect#manage-security-incidents), [Defender for Cloud incident response](/azure/defender-for-cloud/manage-respond-alerts). |
| Exit and Resilience Planning and Drills | • Conduct regular exit rehearsals, disaster recovery tests, and oversight exercises to verify sovereign resilience.<br><br>• If needed, simulate a scenario where the solution is ported off to another environment or in-house, to ensure data and workloads could be extracted if necessary (proving independence from the provider). <br><br>• Perform disaster recovery simulations beyond technical tests - include business continuity and communication aspects. Perform oversight exercises such as internal audits or red team drills specifically targeting sovereign controls (for example, attempt an unauthorized data transfer and ensure it's detected and blocked). <br><br>For more information, see [Business continuity](/azure/cloud-adoption-framework/manage/protect#manage-business-continuity). |
| **Advanced** ||
